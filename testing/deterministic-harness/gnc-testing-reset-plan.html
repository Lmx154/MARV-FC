MARV GNC Cleanup-First Testing Architecture Plan

A single agent-facing implementation plan that combines the deletion-first cleanup plan with the GNC testing architecture plan. The workflow is: clean up and consolidate first, rebuild only necessary proof gates second, then add the new motion-primitive and navigation architecture.

Mode: deletion-first Audience: implementation agent Rule: no version-control operations No placeholders, archives, or wrappers G3 blocked by G2.5 Status date: 2026-05-13 Phase 5 live hardening

1. Decision

Clean first. Consolidate second. Add new architecture third. The agent should remove stale phase-history tests and unused harness helpers before it adds the improved GNC validation architecture.

The current problem is not a lack of tests. The problem is that old tests, synthetic plant checks, phase-numbered files, duplicated helper code, and partially overlapping planning documents make it unclear which tests prove the actual flight-control path. This document replaces that ambiguity with a phased implementation plan.

The agent should treat this file as the canonical work plan once it is placed in the repository. The older GNC refactor and testing-harness planning documents become historical inputs, not competing sources of control flow.

Default action

Delete old tests unless they protect a unique production seam.

Survival rule

A surviving test must be tied to a requirement, a gate, and an actionable failure metric.

Architecture rule

Navigation does not resume until Gazebo motion primitives prove movement, braking, turning, recovery, and estimator agreement.

3. What survives from the two source plans

From the GNC refactor plan

  • Frame safety: truth evidence, estimator-local state, mission waypoints, trajectory setpoints, and control setpoints must not be interchangeable.
  • Takeoff/nav safety: latch estimator XY at takeoff and block navigation until altitude, vertical speed, horizontal speed, yaw health, estimator validity, and clamp ratio are healthy.
  • Controller stability: lateral control is a dt-aware position-to-velocity-to-acceleration cascade with anti-windup.
  • Thrust consistency: lateral acceleration and vertical thrust are planned together so tilt does not silently consume altitude authority.
  • Allocation visibility: saturation must identify which axis lost authority.
  • Gazebo reproducibility: every live gate proves reset cleanliness before measuring flight behavior.

From the testing-harness plan

  • Default CI remains deterministic and offline.
  • Gazebo enters only after deterministic bridge contracts pass.
  • G0 proves physical actuator truth, not only bridge text parsing.
  • G1 uses truth estimate or estimator bypass to isolate controller, mixer/allocation, bridge, frame conversion, and plant behavior.
  • G2 uses real estimator-in-loop hover and vertical behavior before lateral movement.
  • Firmware HIL comes after the portable common control path is validated.

10. Phase 4 — Rebuild the minimal proof ladder through G2

Phase 4

Goal

Bring back only the tests that prove core production seams and basic Gazebo control behavior. Do not work on navigation or G2.5 until C0, G0, G1, and G2 are clear and the report format is stable.

C0 — Contracts

tests/contracts.rs

  • NED/geodetic sign conventions.
  • Quaternion identity/order and finite conversion behavior.
  • Failsafe zero output: invalid estimate, missing IMU, non-finite input, stale input, and disarmed state.
  • Allocator geometry, motor order, saturation/desaturation flags.
  • Bridge parser/formatter/config contracts.
  • HIL duplicate/out-of-order/valid-flag handling.
  • Estimator reset delta finite behavior and setpoint-source tagging.

G0 — Gazebo actuator truth

tests/gazebo_g0_actuator_truth.rs

  • Reset clean before every case.
  • Equal motors do not create large roll/pitch/yaw rates.
  • Motor patterns produce expected angular-rate directions.
  • Hover command produces bounded vertical acceleration.
  • Motor order, spin direction, and scaling are physically proven in Gazebo.

G1 — Control primitives with estimator bypass

tests/gazebo_g1_control_primitives.rs

  • Level hover with bounded motor spread.
  • Roll disturbance recovery.
  • Pitch disturbance recovery.
  • Yaw-rate impulse left/right.
  • Small vertical step up/down while holding XY.
  • Truth estimate is used only through the explicit bypass path.

G2 — Estimator-in-loop takeoff, hover, land

tests/gazebo_g2_takeoff_hover_land.rs

  • Reset clean.
  • Takeoff to controlled altitude.
  • Hover with bounded horizontal drift.
  • Descend and detect near-pad/contact.
  • Disarm to zero motors with no relaunch.
  • Estimator validity and truth agreement are reported.

Phase 4 exit criteria

  • C0 is fast, deterministic, and default-on.
  • G0/G1/G2 are manual or ignored runtime gates until repeatable, but each emits the same JSON/CSV report shape.
  • No phase-numbered compatibility wrappers remain.
  • No navigation tests are active yet.

12. Phase 5 — Add G2.5 motion primitives before navigation

Phase 5

Goal

Prove the vehicle can deliberately move, stop, turn, and recover while maintaining altitude, bounded attitude, speed control, and estimator agreement. These are not waypoint or mission tests.

Navigation remains disabled here. G2.5 uses direct local velocity, acceleration, attitude, or trajectory primitives. Waypoint acceptance and mission state-machine behavior wait for G3.
Control-first boundary. Phase 5 must not depend on compass/GPS mission navigation. The live bridge may expose mag/GPS/truth fields for scoring and estimator comparison, but controller fixes belong in shared control code or firmware-facing control config. IMU/barometer-only stabilization remains the target before G3 navigation resumes.

16. Threshold and failure-policy rules

PolicyRuleReason
Start conservativeThresholds should initially catch obvious failures without requiring perfect tuning.The first goal is reliable layer diagnosis, not final flight performance.
Tighten with evidenceTighten thresholds only after repeated stable runs and clear trace distributions.Prevents fragile tests from blocking structural cleanup.
Never hide lower-gate failuresDo not loosen navigation thresholds if G2.5 primitives still show bad altitude, speed, yaw, or clamp behavior.Navigation should not mask poor low-level control.
Name the first failed layerFailure output must name reset, estimator agreement, guidance/setpoint generation, thrust margin, attitude/rate control, allocation saturation, or plant/bridge behavior.Every failure should produce an actionable next step.

17. Test validity rule

A test is valid only if it passes all four checks:

Production seam

It exercises a seam used by firmware, Gazebo, HIL, or the shared common control path.

Unique failure mode

It catches a failure not already caught by a stronger gate.

Actionable metrics

Its failure says which layer failed and exposes relevant trace values.

No stale surrogate

It does not rely on old synthetic plants or fixtures unless that helper is explicitly under test.

If a test fails any of these checks, delete it. Add it back later only when it can pass all four.

18. What not to do

Do not preserve clutter

  • Do not keep old phase files because they once passed.
  • Do not keep report, fixture, or clock tests unless directly used by current gates.
  • Do not keep synthetic plant tests as acceptance gates.
  • Do not create a hidden graveyard of deleted tests.
  • Do not leave phase-numbered compatibility wrappers in place.

Do not skip motion primitives

  • Do not debug G3 waypoint navigation before G2.5 passes.
  • Do not tune position navigation while straight-line motion is unstable.
  • Do not use navigation success to excuse poor altitude, yaw, speed, or clamp behavior.
  • Do not hide red Gazebo behavior behind looser thresholds.
  • Do not mix truth-origin scoring with estimator-local command frames.

19. Final desired state

The repository should be smaller and more legible after this work. The deterministic harness should have fewer files, fewer helpers, no phase-history naming, and no duplicate host simulations. What remains should be harder to ignore: a compact proof ladder that shows the vehicle can actuate, hover, move, turn, recover, and only then navigate.

Success definition: fewer required tests, fewer duplicate files, one shared report format, one validation matrix, one active plan, and Gazebo failures that identify the failed layer instead of triggering another round of guesswork.
