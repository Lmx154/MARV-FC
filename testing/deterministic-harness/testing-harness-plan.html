MARV Deterministic Test Harness Implementation Plan

An incremental, testable path to validate MARV’s estimator, controller, mixer, actuator path, and Gazebo/HIL integration without debugging the whole flight stack at once.

✓ Host-first ✓ Lockstep time ✓ Component oracles before closed loop ✓ Gazebo only after deterministic tests pass Generated May 11, 2026

1. Objective

Build a deterministic harness that can validate MARV’s flight pipeline in small, observable increments before relying on Gazebo. The goal is not to replace Gazebo; the goal is to make Gazebo failures diagnosable by proving the firmware pipeline in a controlled environment first.

Problem being solved

Gazebo takeoff briefly succeeds, then diverges violently. That can be caused by estimator sign, frame convention, mixer order, motor direction, thrust scaling, stale estimates, or timing.

Validation strategy

Validate each component with deterministic inputs, then integrate components one at a time until closed-loop behavior is proven.

Reference pattern

Use the Holsatus model: host-side simulated hardware, lockstep time, simulated sensors/motors, and an integration test that asserts flight behavior rather than only checking for panics.

Core principle: every increment must produce a binary answer: pass/fail, with logs that identify whether the failure is in sensor ingestion, estimator, controller, mixer, actuator scaling, or external Gazebo integration.

Non-goals for the first version

Non-goalReasonWhen to add later
Photorealistic or full Gazebo physicsToo many variables while debugging firmware logic.After deterministic closed-loop tests pass.
Perfect aerodynamic modelThe first failures are likely frame/sign/order/timing problems, not high-fidelity aero.After hover and setpoint-hold are stable.
Random noise by defaultRandomness makes regressions harder to reproduce.Add seeded noise as explicit test cases.
Full embedded executor fidelityStart by testing pure logic and typed messages.Add a std-device style task runner once the pure harness is useful.

2. Target Architecture

The harness should start as a host-side crate that drives MARV’s portable logic directly. Later increments can wrap the same harness around HIL messages, a std-device firmware runner, and finally Gazebo.

API sketch

pub struct StepInput {
    pub tick: u64,
    pub dt_s: f32,
    pub setpoint: Setpoint,
}

pub struct StepTrace {
    pub tick: u64,
    pub truth: TruthState,
    pub sensors: SensorFrame,
    pub estimate: EstimateSnapshot,
    pub control: ControlSnapshot,
    pub motors: [f32; 4],
    pub clamped: bool,
    pub flags: HarnessFlags,
}

pub trait Plant {
    fn reset(&mut self, initial: TruthState);
    fn sensors(&mut self, truth: &TruthState, tick: u64) -> SensorFrame;
    fn apply_motors(&mut self, motors: [f32; 4], dt_s: f32) -> TruthState;
}

pub trait MarvStep {
    fn step(&mut self, input: SensorFrame, setpoint: Setpoint, dt_s: f32) -> StepTrace;
}
Important design constraint: no test should depend on wall-clock time, thread sleeps, network sockets, or Gazebo until the final optional phases. A failed test must be replayable with the same seed and the same fixture.

3. What to copy from Holsatus, adapted for MARV

Holsatus patternMARV equivalentImplementation requirement
Sim handle exposes vehicle state, motor commands, motor directions, reset/upright, and step. Plant + SimHandle in testing/deterministic-harness. Make the sim state inspectable after every tick. Assertions should never need to infer state from logs only.
Simulated devices implement the same hardware-facing interfaces as real IMU, motors, Vicon, flash. Start with direct MARV data types; later implement HIL semantic routes and actuator sinks. Begin with low-friction pure stepping. Add firmware/task fidelity only after the pure pipeline is validated.
Lockstep executor advances simulated time deliberately instead of sleeping. LockstepClock with advance(dt); all samples derive timestamps from it. Reject duplicate and non-monotonic ticks in tests. Log the exact tick that caused failure.
Integration flight assertion arms, sends setpoints, waits in sim time, and asserts estimated position is near target. Closed-loop cases: static hold, roll disturbance recovery, vertical setpoint, position setpoint. Every closed-loop test must assert both bounded truth state and bounded estimate state.
CI integration runs common tests, embedded builds, and simulated integration flights. Add cargo test -p common, harness tests, then ignored/feature-gated Gazebo tests. Do not add flaky Gazebo to normal CI. Keep deterministic harness in the fast required gate.

Each phase is independently mergeable. Do not move to the next phase until the acceptance tests for the current phase are passing and the evidence artifacts are understandable.

5. Test matrix

This matrix is the “what does this prove?” map. A Gazebo failure should be traced backward through this matrix until the failing layer is identified.

Layer Representative tests Pass proves Does not prove
Pure mixer Mixer equations, clamping, motor-order permutations. Torque command maps to normalized logical/physical motor outputs correctly. Gazebo motor order, motor spin direction, thrust scaling.
Pure controller Zero error, sign basis, altitude NED signs, saturation flags. Controller signs and limits are internally consistent. Estimator latency or sensor frame signs.
Estimator replay Static pad, ascent, dropout, spike, NaN rejection. Estimator handles deterministic sensor streams and remains bounded. Closed-loop behavior under motor-induced motion.
Open-loop plant Motor pulse, roll/pitch/yaw basis response, hover throttle. Actuator geometry and motor-order truth table are correct in the harness. Firmware estimator/control integration.
Closed-loop truth Level hold, disturbance recovery, vertical setpoint. Control + mixer + plant can stabilize when state is perfect. Estimator correctness.
Closed-loop estimator Static hold, vertical setpoint, GPS dropout, baro spike. Full deterministic sensor-estimator-control-actuator loop is sane. Gazebo bridge/config correctness.
HIL semantics Duplicate ticks, stale ticks, response flags, failsafe zero output. MARV’s HIL message/routing layer agrees with the pure harness. Gazebo topic configuration.
Gazebo contract SENSOR parse, ACTUATOR scaling, topic config, motor directions. External sim bridge is not silently changing the contract. Real hardware behavior.

6. Required assertions by phase

Always-on invariants

  • No NaN or Inf in sensors, estimate, controller outputs, motor outputs, or plant state.
  • All accepted ticks are strictly monotonic.
  • All motor outputs are within declared limits.
  • Clamping is explicitly expected or test-failing.
  • Every invalid frame has a documented rejection reason.
  • Every failed test prints the failure tick and last valid trace.

Closed-loop invariants

  • Attitude error decreases after a disturbance.
  • Position or altitude moves toward the setpoint after warmup.
  • Estimator/truth disagreement stays within a configured envelope.
  • Control does not continue with stale or invalid estimate unless explicitly configured to failsafe.
  • Failsafe output is zero or a documented safe value.
  • No sustained saturation during a nominal hover case.
