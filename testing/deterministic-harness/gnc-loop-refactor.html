MARV Flight-Control Refactor: Diagnosis and Phased Implementation Plan

A deep-dive diagnosis of why early tests can pass while live GNC/plant/navigation behavior fails, plus an incremental refactor plan that preserves reusable tests and adds the missing contracts in a logical order.

Target: mixer / estimator / GNC loop Method: PX4 + ArduPilot structural comparison Strategy: incremental gates, not big-bang rewrite Frame: local NED

1. Executive diagnosis

Main conclusion: this does need a significant refactor, but not because the estimator core is obviously wrong. The evidence points first to GNC semantics, takeoff/nav gating, trajectory generation, vertical/lateral thrust coupling, and mixer/control-allocation authority visibility.

The current system already has some good pieces: hardware-independent control modules, local NED conventions, finite-input failsafes, sign-oracle tests, a layered estimator, and deterministic harness gates. The failure is that these pieces are connected in a way that allows unit-correct behavior to become mission-incorrect behavior once the vehicle is airborne and the controller is operating from estimator-relative state.

Early tests passed because they mostly answered, “Are signs, clamps, and nominal outputs plausible?” The failing Gazebo cases ask a harder question: “Can estimator-relative guidance, takeoff settling, lateral trajectory tracking, vertical thrust control, attitude/rate tracking, and physical actuator authority stay mutually consistent over time?” Those are different questions.

2. Evidence-based failure hypotheses

Rank Hypothesis Why it fits the symptoms What would confirm it
1 Setpoint frame semantics are leaking truth into estimator control G3 uses an estimator-driven controller, but the issue ledger says some setpoints were built from truth-origin values. That can make the controller chase a phantom lateral offset even when estimator yaw is fine. A test where truth-origin setpoints fail but estimate-origin/latching setpoints pass with the same estimator output.
2 Takeoff-to-nav transition starts before vertical and horizontal hold are stable Nav started while altitude was not settled. Pre-nav hold used [0,0,takeoff_down], which can generate XY correction if estimator XY is already offset. Nav failures disappear when takeoff latches current estimator XY and waits on altitude error, vertical speed, clamp ratio, and estimator/truth consistency.
3 Position control is too close to direct accel-to-tilt demand The controller has a partial cascade, but no explicit trajectory/path stage and weak bias handling. The integral path has no explicit dt in the update, so tuning is loop-rate dependent unless gains are treated as per-tick. Jerk/velocity-limited setpoint generation and dt-aware velocity PID reduce cross-track swings without raising position gain.
4 Mixer hides authority loss behind a single clamped flag The mixer algebra clamps normalized commands, but does not expose roll/pitch/yaw/throttle limit flags, desaturation, axis priority, physical geometry, spin direction, or thrust/drag coefficients in the common control path. Logs show yaw/rate/motor clamp during lateral demand, and new axis-specific saturation flags explain which axis lost authority first.
5 Estimator interface reset handling is missing, not necessarily estimator math The estimator architecture is structurally plausible, but controller setpoints and filters must handle local-position, z, velocity, and heading resets. PX4 explicitly adjusts setpoints/filters on estimator resets. Injected estimator origin/heading/z reset tests show either a jump in control demand or a latched compensation path.
6 Gazebo reset/initial-state contamination is corrupting some tuning probes The issue ledger mentions later probes with extreme pre-nav drift and estimator/truth disagreement. That can make gain studies lie. Hard reset + settle + zero-energy assertions produce repeatable initial traces before every ignored Gazebo gate.

3. Known-good structure to converge toward

Established autopilots do not usually treat “go to this waypoint” as a direct position-error-to-tilt calculation. They separate estimator state, navigation/path generation, position/velocity control, thrust/attitude setpoint construction, attitude/rate control, and actuator allocation.

Sensors ↓ Estimator / local NED state ↓ Mode + Guidance state machine ↓ Trajectory / setpoint generator - position target - velocity target - acceleration feed-forward - yaw / yaw-rate target ↓ Position controller - position P → velocity target - velocity PID + feed-forward → acceleration / thrust vector - dt-aware integrators and anti-windup ↓ Attitude setpoint builder - thrust vector + yaw → attitude quaternion + collective thrust ↓ Attitude controller - quaternion error → body-rate setpoint ↓ Rate controller - body-rate PID → torque + thrust demand ↓ Control allocation / mixer - physical geometry - motor order - spin direction - thrust/drag coefficients - desaturation and axis-specific limit flags ↓ Actuator outputs

What MARV already has

  • Portable control modules: position, altitude, attitude, rate, mixer.
  • Local NED conventions and sign-oracle tests.
  • Finite-input rejection and zero-output failsafe.
  • A layered estimator with attitude ESKF and navigation EKF.
  • Deterministic host and Gazebo harness phases.

What is missing or too thin

  • Explicit guidance/trajectory layer with path semantics.
  • Estimator-origin setpoint contract and reset compensation.
  • Takeoff/nav transition latch and health gates.
  • Unified lateral+vertical thrust vector generation.
  • dt-aware integral behavior and saturation feedback.
  • Physical control allocation and axis-specific motor limits.

4. Why the beginning tests could pass

Sign tests are necessary but weak

They catch reversed axes and invalid motor equations, but they do not prove that a mission-level trajectory, estimator origin, vertical thrust, and motor authority are consistent.

Demand-limit tests prove clipping, not tracking

A clipped demand can be safe and still be dynamically unable to follow a waypoint. The controller can remain finite while being under-actuated or saturating the wrong axis.

Simplified navigation tests skip live dynamics

The pure navigation step test integrates horizontal acceleration inferred from commanded tilt. It does not exercise real motor lag, rate tracking, vertical thrust loss from tilt, yaw torque coupling, or actuator saturation.

Estimator replay can be too ideal

Estimator-in-loop tests can pass if sensor fixtures and plant coupling are too clean. Live Gazebo adds resets, initial conditions, timing, frame conversions, and actuator effects.

The mixer only says “clamped”

A single clamped boolean does not reveal whether altitude, yaw, roll, pitch, or all axes lost authority first.

Controller state is incomplete

A path follower needs memory: active leg, origin latch, velocity/accel setpoint, reset counters, and transition state. A mostly stateless position step can pass one-tick tests and fail over a mission.

Important: each phase should keep the previous tests passing unless the phase intentionally changes a contract. When a contract changes, first write a legacy-vs-new comparison test, then change the assertion. That prevents a refactor from becoming a tuning expedition.

8. Concrete acceptance criteria

  • Frame safety: truth-origin data cannot enter the control setpoint path except through an explicitly named conversion used only in tests.
  • Takeoff safety: lateral hold latches estimator XY; nav cannot start until altitude, vertical speed, estimator validity, and clamp ratio are healthy.
  • Controller stability: position integral behavior is dt-aware, bounded, and anti-windup protected.
  • Thrust consistency: lateral tilt and vertical thrust are planned together; altitude authority is not silently consumed by lateral demand.
  • Allocation visibility: output limits report which axis saturated and whether roll/pitch, yaw, or throttle was prioritized.
  • Estimator reset handling: local-position, velocity, z, and heading resets do not cause a sudden mission-level command jump.
  • Gazebo reproducibility: every live gate first proves reset cleanliness before measuring control behavior.

9. Near-term patch order

  1. Add debug fields and axis-specific diagnostic counters without changing control behavior.
  2. Patch takeoff latch and nav gate. This is the highest signal-to-risk change.
  3. Type-separate truth evidence from estimator-local command frames.
  4. Move waypoint slew/path generation into control/guidance.
  5. Refactor lateral position into dt-aware pos→vel→accel cascade.
  6. Unify vertical acceleration and lateral acceleration into a thrust-vector setpoint.
  7. Replace logical mixer with physical allocator behind a compatibility wrapper.
  8. Add estimator-reset synchronization and Gazebo reset-cleanliness gates.

10. Bottom line

The current architecture is not “terrible,” but it is too shallow in the exact layers that make a multicopter mission robust: frame semantics, guidance/path state, transition gates, thrust-vector construction, control allocation, and saturation feedback. The estimator core looks closer to a sane structure than the GNC/mixer path does. Refactor the GNC and allocation seams first, keep estimator math stable unless the reset/interface tests prove otherwise, and use every phase to convert a suspected failure mode into a reusable test.
